# 実装計画 - receipt_new_design

## 目的
既存の「Aさん→Bさんへ、x円払う」という最終結果だけでは内部計算の理解が難しいため、
「詳細な計算フロー」画面をメンバー単位でより直感的に把握できるUIへ変更する。

## 変更方針
1. 精算計算ロジックの再利用性を高める。
   - 精算項目ごとの負担内訳（誰がいくら負担したか）を `calculateExpenseBreakdowns` として抽出。
   - 最終精算（送金）と明細表示が同じ計算結果を参照する構成にする。
2. 詳細UIをメンバー中心に再設計する。
   - 複数選択フィルタを設置し、表示対象メンバーをユーザーが切り替え可能にする。
   - 各メンバーに対して「支払う額」「受け取る額」を別カードで表示する。
   - その下に「トータル結果」カードを置き、最終的な支払い/受け取り関係を表示する。
3. 視認性ルールを反映する。
   - 支払う金額は赤系、受け取る金額は緑系。
   - トータル結果では `→`（支払い）と `←`（受け取り）を明示する。

## 追記: 比率入力ロジック改善
- `onBlur` 時に、0または空欄の入力を「未編集扱い」に戻して他メンバー割合から再補完する。
- 再計算結果に0%メンバーが含まれる場合は、全アクティブメンバーが最低1%となるように自動再分配する。
- 自動再分配が発生した際はトーストで警告を表示する。
- 勾配モードに「均等化」ボタンを追加し、アクティブメンバー比率を合計100%で均等再分配できるようにする。
- 勾配入力フォームの編集状態を視覚的に明確化するため、色差分に加えて「変更済み」ラベルを表示する。
- 勾配比率は入力中の自由度を維持しつつ、`onBlur` と保存時に合計100%へ自動補正する。
- 補正不能なケースでは保存を停止し、ユーザーに再入力を促す。
- ホーム「メンバー」の管理アクションの視認性を改善し、押せるUIであることを明確化する。
- 精算項目セクションのヘッダー右側に、登録画面へ直接遷移する「登録する」ボタンを追加する。
- メンバー登録時に同名チェックを追加し、重複時はトーストで警告して追加を拒否する。
- メンバー管理画面の各行に「名前変更」操作を追加し、インライン編集で保存できるようにする。
- 非勾配モード保存時に比率値を `1` へ強制上書きしていたため、保存値の扱いを見直す。
- 計算側で `isGradientMode=false` のときだけ比率を実効的に `1` とみなすように変更し、保存データと表示データの整合を保つ。
- 勾配比率入力を小数第2位まで受け付けるようにし、内部では固定スケール整数計算で誤差を抑制する。
- 精算項目リストの対象表示をテキスト件数から、誰が対象か分かるアイコン表示へ変更する。
- 立替者/端数担当者が視認しやすいよう、対象順序とバッジ表示を調整する。
- 対象メンバーが多い場合に備え、8人単位で段階表示できる「もっと見る」UIを追加する。
